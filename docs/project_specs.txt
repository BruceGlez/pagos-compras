Overview

This project is an internal business system designed to automate the workflow used to manage cotton purchase payments.

The company purchases cotton bales from producers. Each purchase must go through a series of operational, accounting, and administrative steps before the producer is paid.

Currently this workflow is performed manually using multiple systems:

algodon.net (desktop system used to record cotton purchases)

Excel spreadsheets used to track the payment pipeline

Microsip ERP (Firebird database) used to track producer debts

Gmail for invoice communication with accountants

SAT portal for validating invoices (CFDI)

BBVA NetCash for executing payments

Windows file server for storing digital documents

The goal of this project is to replace the manual Excel pipeline and centralize the entire workflow into a single internal system.

The system should manage the full lifecycle of a cotton purchase from import to final payment and digital expediente completion.

Business Workflow

The current operational workflow is:

Producer sells cotton bales
→ Purchase recorded in algodon.net
→ Purchase PDF saved to server
→ Purchase manually entered into Excel tracking sheet
→ Debt checked in Microsip
→ Invoice requested from accountant
→ Accountant sends invoice (XML + PDF)
→ Invoice validated against SAT and internal rules
→ Bank account confirmed with producer
→ Payment executed in BBVA NetCash
→ Payment receipt downloaded
→ Documents saved in server folder
→ Excel updated
→ Receipt sent to producer

This entire process must be automated or partially automated.

Key Business Concepts
Purchase (Compra)

A purchase represents cotton bales bought from a producer.

Each purchase contains information such as:

producer

number of bales

weight in pounds

total value

currency

tax regime

season

Purchases may sometimes be divided into multiple sub-purchases called divisions.

Each division may:

use a different RFC

generate a separate invoice

generate a separate payment

have its own expediente

However divisions may still belong to a parent purchase.

Debt (Deudas)

Before paying a producer, the company subtracts any debts owed by that producer.

Debt is retrieved from Microsip ERP.

Debt rules:

all open balances count as debt

overdue and non-overdue balances count

debts from current season and prior seasons count

debt is deducted from the purchase payment

However, sometimes debt exists outside the ERP system.
The operator must be able to add manual deductions.

Payment Calculation

The amount paid to the producer is calculated using the following formula:

saldo_a_pagar =

purchase_total
minus anticipos (advance payments)
minus microsip debt
minus RESICO retention (1.25%)
minus manual deductions

Manual deductions may include:

coberturas

external loans

accounting adjustments

Invoice (Factura)

Before payment can be made, the producer must issue a valid invoice (CFDI).

Invoices are normally generated by the producer’s accountant.

Invoices are typically received by email in the form of:

XML file
PDF file

Invoice Validation

Invoices must pass validation before payment.

Required checks include:

correct RFC

correct producer name

uso CFDI = adquisición de mercancías

correct payment method

correct currency

IVA rate = 0%

If the producer is in the RESICO regime:

the invoice must include ISR retention 1.25%
OR

include the legal exemption legend

Invoices that fail validation must block payment.

Bank Confirmation

Before executing payment, the operator must confirm the bank account with the producer.

Even if bank data exists in the system, confirmation is normally requested through WhatsApp.

The system should track:

bank account used

confirmation timestamp

confirmation source

Payment Execution

Payments are executed using BBVA NetCash.

Initially the system will not execute payments automatically.

Instead it will:

prepare payment information

allow operator to perform payment manually

record payment details

store the payment receipt

Payment Priority

Payments are not strictly processed in chronological order.

Priority depends on:

available cash

nearing the 15-day payment promise

producer urgency

manual override by management

The system should compute a priority score for each purchase.

Digital Expediente

Each purchase must maintain a digital expediente containing all documents.

Typical expediente contents:

compra_base.pdf
compra_division.pdf
factura.xml
factura_pdf_proveedor.pdf
sat_pdf_validado.pdf
comprobante_pago.pdf
rep_ppd.pdf

Files should be stored on the company file server.

The system should track file locations in the database.

Purchase Workflow States

Purchases should move through the following states:

IMPORTED
DEBT_CALCULATED
WAITING_INVOICE
INVOICE_RECEIVED
INVOICE_VALID
INVOICE_BLOCKED
WAITING_BANK_CONFIRMATION
READY_TO_PAY
PAID
ARCHIVED

State transitions should be logged.

System Integrations
algodon.net

No API available.

Purchases will be imported from Excel export files.

Microsip ERP

Microsip uses a Firebird database.

The system should query the database to retrieve producer debt balances.

Gmail

Invoices arrive via email.

Future automation may:

scan inbox
extract attachments
parse XML

BBVA NetCash

Payments executed manually.

System records payment details and receipts.

Design Principles

Use explicit workflow states instead of Excel columns.

Keep business logic inside backend services.

Maintain full audit trail of operations.

Store all documents in digital expediente folders.

Build features incrementally to avoid large rewrites.

Payment execution should remain manual initially for safety.

Expected Outcome

The operator should no longer need to manually coordinate multiple systems.

Instead of opening Excel, Microsip, email, SAT, and bank interfaces separately, the operator will see a single dashboard displaying:

purchases waiting invoice
blocked invoices
payments ready
payments completed

The operator will only handle exceptions while the system manages the workflow automatically.
